# 3D 引擎与 UI 状态解耦与同步规范

为了在维持 60 FPS+ 极致渲染性能的同时保证 UI 展现层的响应式更新，WebCraft 的 3D 游戏引擎（基于 `requestAnimationFrame` 驱动的渲染、物理更新与输入检测）与 React UI 必须严格遵循以下解耦与同步架构规范：

---

## 1. 状态分类与存储原则

### 引擎内部高频状态（严格隔离）
* **定义**：玩家实时坐标、每一帧的相机变换与视口向量、物理碰撞中间量等。
* **规范**：**严禁**将此类高频更新的数据直接存入 Zustand 等 React 状态库或触发 React 重绘，它们必须完全保留在 3D 引擎 of 内存中，以防止频繁触发 React 重绘导致的严重掉帧。

### 低频全局状态（Zustand 托管）
* **定义**：游戏生命周期阶段（主菜单、游戏运行中、暂停等）、玩家当前选中的方块类型、玩家受击红幕特效等离散事件、游戏配置（FOV 视场角、渲染距离等）。
* **规范**：统一通过状态管理库（Zustand）同步，以便 React 能够响应式地渲染最新 UI。

### 监控/诊断指标（降频同步）
* **定义**：需要在 UI 实时呈现的性能指标，如 FPS、物理实体总数等。
* **规范**：引擎必须采用降频节流（Throttle）机制定时同步至状态库，禁止逐帧无节流同步。

---

## 2. 读写与订阅规范

### UI 响应式订阅（细粒度）
* React 组件订阅全局状态时，**必须使用 Selector 模式**。
* 示例：
  ```typescript
  // 推荐：仅在 fov 发生改变时组件才会重绘
  const fov = useGameStore(state => state.settings.fov);
  
  // 杜绝：这会导致状态中任意属性变化都引起组件重新渲染
  const state = useGameStore(); 
  ```

### 引擎非响应式读取（直接获取）
* 3D 引擎在主循环或物理更新等高频逻辑中，如果需要访问配置或状态，**严禁使用 React Hook**。
* **规范**：必须使用非响应式接口直接读取当前状态快照。
* 示例：
  ```typescript
  // 推荐：直接获取当前状态快照，不触发任何 React 重绘与 Hook 依赖
  const fov = useGameStore.getState().settings.fov;
  ```

### 单向事件驱动写入（主动推送）
* 游戏引擎与 React UI 之间的状态同步应当是**事件驱动的单向流动**。
* **规范**：仅在发生离散的用户操作（如设置修改）、生命周期跃迁或重大属性变更（如受击伤害）时，才主动调用 Store 的 Action 触发同步。
