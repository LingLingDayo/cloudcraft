# 环境与天气系统设计规范

> [!NOTE]
> 该功能模块对应的源码路径为 `src/game/environment/`。

为了在提供逼真的多维度与多天气动态视觉体验的同时，保持高频渲染管线的流畅性与扩展性，CloudCraft 的环境与天气渲染系统必须严格遵循以下架构设计规范与表现契约：

---

## 1. 核心设计原则与分层

环境与天气系统应当遵循严格的职责分工，划分为三个层级：

1. **维度与配置定义（静态层）**
   * 负责存储不同维度（如主世界、下界、洞穴）的天空基准色、雾浓度、光照基准强度等静态配置。
   * 配置应当集中管理，通过开关决定当前维度是否启用昼夜交替周期或采用固定时间。
   * 关联文件位置：`src/game/environment/DimensionConfig.ts`

2. **状态与时间驱动（逻辑层）**
   * 维护游戏当前的运行时间周期（用以计算天体的位置角度）。
   * 维护当前天气状态的过渡权重（在晴天、雨天、暴风雨等天气类型间进行平滑的线性或曲线混合）。
   * 负责检测相机环境状态（例如是否处于液体等介质中），以驱动渲染参数的改写。
   * 关联文件位置：`src/game/environment/EnvironmentState.ts` 与 `src/game/environment/WeatherPresets.ts`

3. **表现与渲染应用（表现层）**
   * 负责插值计算，将多个天气预设的混合结果合成最终的渲染参数，并单向应用至 Three.js 的场景（背景清理色、雾色、雾浓度）和灯光中。
   * 负责天体的轨迹运行和阴影投射管理。
   * 关联文件位置：`src/game/environment/WeatherBlender.ts`、`src/game/environment/EnvironmentRenderer.ts` 及 `src/game/environment/CelestialBodies.ts`

---

## 2. 架构级方向约束

以下为系统完全确定的最基本定性事实与设计逻辑，任何代码改动都必须遵守这些架构约束：

* **视觉一致性契约**：为了防止远景地形在天空中形成突兀的剪影，天空背景的清理颜色（Clear Color）必须始终与场景雾颜色（Fog Color）保持对齐。这是确保远距离渲染无缝契合的视觉表现规范。
* **介质渲染绝对优先级**：相机所处的局部物理介质（如水体等液体）具有最高的渲染覆盖优先级。一旦检测到相机进入该介质，渲染系统必须立刻应用介质专属的雾效与色调，覆盖常规的维度和天气天空色配置。
* **状态单向流向表现**：由时间流逝和天气变化派生出的环境参数是一条单向传递的管道（`状态 -> 混合 -> 渲染应用`）。禁止在渲染器中反向修改时间状态或天气权重。

---

## 3. 开发与扩展方向引导

### 如何新增维度或天气？
* **新增维度**：在维度配置中定义全新的维度配置，提供其基准天空色彩与雾效参数，并指定是否随时间流逝。
* **新增天气**：实现天气接口定义，配置 24 小时随时间演变的插值关键帧，并在混合器中注册。

### 零 GC（垃圾回收）性能约束
* **高频 Tick 零运行时分配**：由于环境计算与天体运动属于 60 FPS 渲染主循环的一部分，**绝对禁止**在每帧调用的更新、混合与渲染逻辑中临时实例化新的颜色（Color）、向量（Vector3）或矩阵对象。
* **做法**：所有数学辅助计算和临时混合数据必须复用预先分配并在类实例化时初始化好的私有成员变量，通过引用改写（如 `.copy()`、`.set()` 或 `.lerp()`）进行运算。
