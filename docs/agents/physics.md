# 物理与碰撞系统开发规范

本文档明示了在对 Minicraft 的物理、移动、碰撞系统进行修改、重构或功能添加时，必须严格遵守的架构契约与设计约束。

---

## 🚨 核心开发红线 (设计约束与避坑指南)

### 1. 分轴更新与检测原则 (AABB Voxel Collision)
在修改 `VoxelPhysics.ts` 中的移动逻辑时，**严禁一次性将所有轴向的速度加到位置上**。
* **开发约束**：必须按轴（X 轴、Z 轴、Y 轴）独立应用速度，并立刻进行碰撞判定与位置回退/对齐。
  * 示例规范：
    1. 先在 X 轴上应用速度，判断 X 轴碰撞并解决（对齐或速度归零）。
    2. 再在 Z 轴上应用速度，判断 Z 轴碰撞并解决。
    3. 最后在 Y 轴上应用速度，处理跳跃/重力下落并解决。
* **避坑警告**：若合并轴向进行一次性判定，会导致实体在顶角、斜对角移动时发生穿透或剧烈抖动，破坏沙盒移动手感。

### 2. 公共接口封装与稳定性约束 (API Design & Stability)
* **最小暴露原则**：`Physics.ts` 作为物理系统的统一管理器入口，必须实现对内（自研体素物理与局部刚体物理）的封装。对外仅暴露实体大小、重力配置、液体环境检测与步进 Tick 等最小接口集，具体的碰撞算法与中间计算逻辑应完全内聚于子模块中。
* **变更兼容性约束**：严禁在未评估全局调用链的前提下，随意修改或破坏 `Physics.ts` 的公共方法签名。接口变更必须以向后兼容为首要考虑，以防止引发大范围的外部依赖模块（如 `GameManager` 或 React UI 组件）的构建连锁崩溃。
* **接口变更测试同步**：任何公共接口参数或行为的调整，必须第一时间重构并更新对应的单元测试（如 `Physics.test.ts`）与测试 Mock 实现，以保障 CI/CD 流程的稳定性。

### 3. 高频状态严格隔离规约
物理计算处于每帧更新的高频循环（60FPS+）中。
* **开发约束**：**严禁在物理主循环的每 Tick 中向 Zustand 状态库（useGameStore）直接推送任何高频物理量（如玩家实时坐标、精确速度、碰撞状态）**。
* **同步规范**：如需在 UI 呈现（如显示坐标或调试面板），物理系统必须使用限频节流（Throttle，如每 100ms 一次）将数据批量同步至状态库。

### 4. 局部辅助刚体集成约束 (Rapier.js Integration Bounds)
若实现复杂抛体、载具或爆炸碎屑等高级刚体仿真，需要集成 Rapier.js。
* **开发约束**：**严禁将整个沙盒世界的体素方块全量导入 Rapier 作为静态 Collider**。
* **优化策略**：必须采用**动态局部 Collider 注入机制**。根据运动实体的坐标，动态判定周边仅 $3 \times 3 \times 3$ 的固态方块，将其注册为静态 Collider，超出范围的即时释放。
* **职责划分**：WASM 辅助物理引擎仅处理局部抛体和碎屑，严禁替代自研体素物理去托管核心角色的移动控制。

---

## 🧪 单元测试守护规范

在对任何物理计算或碰撞检测逻辑进行修改后：
1. **测试同步更新**：必须首先检查 `src/game/physics/Physics.test.ts`，如果修改了判定规则或变量（如重力常数），必须同步修复测试脚本中的 Mock 数据。
2. **测试运行**：确保在提交之前在本地运行 `npm run test`。所有物理测试用例必须 100% 通过。
