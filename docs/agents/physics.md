# 物理与碰撞系统设计及开发规范

> [!NOTE]
> 该功能模块对应的源码路径为 `src/game/physics/`。

---

## 1. 物理系统核心架构

物理系统采用 **“自研体素物理主导 + 局部刚体物理引擎辅助”** 的混合架构：
* **Physics** (`src/game/physics/Physics.ts`): 物理系统的统一对外接口与生命周期 Tick 管理。
* **VoxelCollider** (`src/game/physics/voxel/VoxelCollider.ts`): 基于 AABB 的体素几何碰撞判定与方块检索。
* **CharacterController** (`src/game/physics/voxel/CharacterController.ts`): 控制动态实体（如玩家）的移动流、台阶步进、自动跳跃及状态刷新。
* **MovementStrategy** (`src/game/physics/voxel/strategies/`): 游泳、行走、飞行等策略接口与解耦实现。

---

## 2. 核心算法执行时序

运动控制器在处理动态实体的每 Tick 位移时，必须遵循以下执行时序：
1. **流体检测**：根据 AABB 获取流体环境，刷新受力速度。
2. **分轴碰撞检测与位移更新**：
   * **X轴**：应用位移 -> 碰撞判定 -> 回退位置 -> 速度归零。
   * **Z轴**：应用位移 -> 碰撞判定 -> 回退位置 -> 速度归零。
3. **台阶步进 (Step-up)**：在 X/Z 轴碰撞受阻时，尝试虚拟向上抬升（Step Height）并在新高度重算 X/Z 位移。若水平位移优于平地碰撞，则应用抬升位置，否则回退并判定是否触发自动跳跃。
4. **Y轴**：应用位移 -> 碰撞判定 -> 回退位置 -> 速度归零（判定触地 `onGround` 或撞头）。

---

## 3. 开发红线与设计约束 (Agent 必读)

### 3.1 严格分轴判定，严禁合并轴向
* **红线**：**绝对禁止一次性将三轴的位移或速度合并进行碰撞判定。**
* **理由**：合并判定会导致顶角或斜对角移动时无法区分碰撞法线，造成穿墙或抖动。必须严格按 X -> Z -> (台阶判定) -> Y 顺序依次计算。

### 3.2 引擎高频状态与 UI 库绝对隔离
* **红线**：**严禁在高频 Tick 循环 (60FPS+) 中直接推送高频状态（如玩家精确坐标、即时速度）到 Zustand 等 React UI 状态库。**
* **规范**：调试指标必须使用节流（如 100ms 间隔）或 `requestAnimationFrame` 降频同步，防止 React 重绘导致卡顿。

### 3.3 局部刚体集成限制 (如 Rapier 引擎)
* **红线**：**严禁将整个沙盒世界的体素方块全量注册进物理引擎。**
* **规范**：必须以刚体实体为中心，仅动态注入周围（如 $3\times3\times3$）的体素作为静态碰撞体，随实体移动动态销毁。辅助引擎仅用于抛射物或爆炸粒子等局部刚体仿真。

---

## 4. 单元测试与验证规范

* **测试同步更新**：修改物理计算或碰撞规则时，必须同步更新 `src/game/physics/` 同级目录下的 `.test.ts` 文件。
* **本地验证命令**：
  ```bash
  npm run test:run -- src/game/physics
  ```
* **通过契约**：所有测试用例必须 100% 通过，且无任何 TypeScript 类型校验错误。
