# 测试脚本同步更新与验证规范

* **测试识别**：在修改任何功能、核心逻辑（如 `src/game/` 内的核心计算）或状态管理（`src/store/`）时，必须主动检索并识别是否有对应的测试脚本或与源文件同级的 `.test.ts` 文件）。
* **同步更新**：如果修改的业务或核心逻辑已有对应测试，**必须同步更新、修复或重构对应的测试用例**，确保测试能真实并正确地覆盖最新的代码行为，严禁直接忽略或跳过受影响的现有测试。
* **新增测试**：若引入了新的核心算法（如新的噪波算法、数学计算工具、数据序列化逻辑等），必须在 `tests/` 或同级目录下编写并提交对应的单元测试，若测试中包含 `new World(seed)`，seed 字符串中必须包含 `TEST_SEED_IDENTIFIER` 标识值（即包含 'test' 关键字），以确保其能被识别为测试种子并自动激活测试性能优化机制。
* **执行验证**：在完成代码修改与测试脚本更新后，**必须**按以下规则在本地运行测试并确保全部通过，保证无回归缺陷（Regression）：
  * **重构场景**（如目录结构调整、模块拆分、接口变更等影响范围较广的改动）：**必须全量运行** `npm run test:run`，确保整个项目无回归；
  * **常规修改场景**（新增功能、局部 Bug 修复等）：运行**受影响模块的目录级测试** `npm run test:run -- path/to/module`；
  * **禁止单文件测试**：任何情况下禁止将测试范围缩小至单个文件，最小粒度为**模块目录**。
