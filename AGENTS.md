本指南旨在明确 CloudCraft 项目的设计思想、全局开发规范与流程，以确保开发贡献与协作过程中，能够编写出高质量、高可维护性的代码。

> [!IMPORTANT]
> **分布式模块开发契约 (Mandatory Rule for Agents)**:
> 1. **先读后写**：在开发或修改某个特定模块的功能前，**必须首先阅读**该模块主路径下的 `README.md`（各模块文档索引参见 [docs/modules.md](docs/modules.md)），明确其设计规范与开发红线。
> 2. **同步更新**：在完成代码修改后，**您必须根据最新的代码逻辑和接口行为，同步更新该模块下的 `README.md`**，以保持文档与代码的一致性。

## 1. 项目概述与目标

### 什么是 CloudCraft？
CloudCraft 是一个运行在浏览器中的 3D 沙盒游戏，旨在模仿经典的 Minecraft（我的世界）。该项目利用现代前端工程化方案和 WebGL 的优势，将 React 的声明式状态管理能力与 Three.js 的 3D 渲染性能相结合，提供流畅的游戏体验。

### 项目核心目标
* **极致性能**：在浏览器端流畅运行（目标 60 FPS+），合理管理 Three.js 的内存、网格合并与渲染管线。
* **清晰的系统分层**：解耦 3D 游戏物理/渲染层与 React UI 展现层。
* **完整的沙盒体验**：包含随机地形生成（基于噪波算法）、方块的放置与破坏、基于 AABB 的碰撞检测与重力物理、动态粒子反馈及合成音效系统。
* **工程化规范**：严格 TypeScript 类型约束、基于 Sass Modules 的模块化样式、严谨的单元测试体系。

---

## 2. 技术栈架构

* **核心框架**：React 19 & TypeScript & Vite
* **3D 渲染**：Three.js
* **状态管理**：Zustand (实现 UI 与 3D 游戏状态的低频高效同步)
* **样式方案**：Sass / Scss Modules
* **测试框架**：Vitest & Testing Library

## 3. 开发与编码规范 (全员必读)

在对本项目进行修改或增加功能时，必须严格遵守以下开发与编码规范：

**目录组织与架构**：新增或调整代码时，**必须将当前项目视为大型工程来进行组织**，编写高内聚、低耦合的代码，明确划分模块职责，以方便后期的高效拓展与长期维护。当代码行数接近或超过 600 行时，必须进行重构与拆分。应当将逻辑和视图拆分为更小粒度的子组件或独立的函数/工具模块，并组织在清晰、合理的目录结构中。

**状态管理**：全局状态统一通过 `src/store/useGameStore.ts` 的 slices 访问，禁止在组件层直接维护跨组件共享状态。

**样式编写规范**：本项目使用原生的 Sass 模块化开发。所有新添加 React 组件都应当在同目录下创建对应的 `*.module.scss`，并使用 CSS 变量来继承全局色彩主题。

**TypeScript 强类型**：
* 所有新函数、新组件属性都必须编写完备 TypeScript 类型。
* 在与 Three.js 交互时，避免使用 `any` 类型，严格使用 Three.js 的原生数学类（Vector3、Euler）和三维对象类型。

**注释规范**：
* 所有的核心函数、复杂业务逻辑和关键功能**必须书写简明扼要的注释**。

**计算性能与多线程分工**：
* 严禁在主线程执行高延迟、高开销的 CPU 密集型计算（如高频噪波计算、大范围方块遍历、复杂的网格剖分/Greedy Meshing 合并等），以防止主线程卡顿掉帧。
* 所有复杂几何计算和数据填充必须移至 Web Worker 多线程中异步运行，并使用 Transferable Objects 实现 TypedArray 的零拷贝传递，确保主线程交互的绝对顺畅。

**避免硬编码与重复配置**：
* 严禁在代码中直接写魔法数字（Magic Numbers）、硬编码的颜色值、时长、物理属性等。
* 所有配置项必须存放在对应模块的域内配置文件中。
* 严禁复制粘贴相同的生成逻辑或概率计算，必须提取统一的通用算法函数与对应的配置结构。

**Lint和类型检查**：修改代码后，**必须**运行 `eslint`、`build` 和 `tsc` 校验并修复所有问题。严禁仅凭 IDE 提示就宣布任务完成。

**测试脚本同步更新与验证**：相关规范参见 [docs/testing.md](docs/testing.md)。当修改代码或需要验证修改时，必须阅读该文件以获取具体的测试和验证要求。

**临时文件管理**：所有生成的临时文件、调试脚本或中间产物必须存放到 `.temp/` 文件夹。
