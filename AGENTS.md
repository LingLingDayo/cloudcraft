# WebCraft 开发指南与规范总纲 (AGENTS.md)

本指南旨在明确 WebCraft 项目的设计思想、全局开发规范与流程，以确保开发贡献与协作过程中，能够编写出高质量、高可维护性的代码。

本项目采用**渐进式披露（Progressive Disclosure）**方案管理文档：
* **AGENTS.md**（本文件）保存最通用的开发与编码规范，为所有任务必读之总纲。
* **特定领域的架构规范与设计约束**已拆分为独立专题文档，您可以在 [## 4. 核心设计原则](#4-核心设计原则-概述与引导) 处查阅各个专题的概要与直达链接。

## 1. 项目概述与目标

### 什么是 WebCraft？
WebCraft 是一个运行在浏览器中的 3D 沙盒游戏，旨在模仿经典的 Minecraft（我的世界）。它利用了现代前端工程化方案和 WebGL 的优势，将 React 的声明式状态管理能力与 Three.js 的 3D 渲染性能相结合，提供流畅的游戏体验。

### 项目核心目标
* **极致性能**：在浏览器端流畅运行（目标 60 FPS+），合理管理 Three.js 的内存、网格合并与渲染管线。
* **清晰的系统分层**：解耦 3D 游戏物理/渲染层与 React UI 展现层。
* **完整的沙盒体验**：包含随机地形生成（基于噪波算法）、方块的放置与破坏、基于 AABB 的碰撞检测与重力物理、动态粒子反馈及合成音效系统。
* **工程化规范**：严格 TypeScript 类型约束、基于 Sass Modules 的模块化样式、严谨的单元测试体系。

---

## 2. 技术栈架构

* **核心框架**：React 19 & TypeScript & Vite
* **3D 渲染**：Three.js
* **状态管理**：Zustand (实现 UI 与 3D 游戏状态的低频高效同步)
* **样式方案**：Sass / Scss Modules
* **测试框架**：Vitest & Testing Library

## 3. 目录结构与模块划分

```text
src/
├── assets/          # 静态资源（纹理贴图、音效等）
├── components/      # React UI 组件
│   ├── common/      # 通用基础组件（按钮、滑块等）
│   └── views/       # 游戏页面视图（StartMenu, HUD, PauseMenu）
├── game/            # 3D 游戏引擎核心逻辑 (纯 TypeScript)
│   ├── core/        # GameManager (引擎主控与生命周期)
│   ├── entities/    # 实体系统 (Player 等)
│   ├── physics/     # 物理系统 (AABB 碰撞检测、重力)
│   ├── systems/     # 辅助系统 (Controls 输入绑定, Particles 粒子, Sound 音效)
│   └── world/       # 世界与地形生成 (World, Noise 噪波, TextureAtlas 纹理合批)
├── store/           # Zustand 状态仓库 (slices 模式：game, player, debug)
├── styles/          # 全局 Sass 样式与主题定义
├── types/           # 共享的 TypeScript 类型声明
├── main.tsx         # 应用挂载入口
└── App.tsx          # 顶层应用组件
```

## 4. 核心设计原则 (概述与引导)

### 3D 引擎与 UI 状态同步
* 引擎的渲染与物理循环处于高频 Tick（60 FPS+）中，而 React UI 则处于低频渲染中。
* 为了避免频繁重绘带来的卡顿，必须严格解耦高频状态和全局低频状态。
* 🔍 **详细规范与最佳实践**，请参阅：[docs/agents/state_sync.md](docs/agents/state_sync.md)。

### 物理与碰撞系统
* 游戏的核心移动与重力通过自研的 AABB 体素物理系统控制，采用分轴检测以确保流畅和墙体滑动效果。
* 未来的复杂物理仿真（如弹射物、吊桥等）将使用 Rapier.js 引擎作为局部辅助。
* 🔍 **物理系统的设计细节与集成步骤**，请参阅：[docs/agents/physics.md](docs/agents/physics.md)。

### 区块生成管道流水线
* 世界生成器使用分阶段的管道流水线生成区块数据（包括高度图计算、地形填充、洞穴雕刻、地表装饰与树木生成）。
* **重要安全契约**：如果需要新增或修改任何地形、洞穴、树木或地表植被的生成逻辑，**必须**先阅读此文档以确保流水线规范及生成逻辑的一致性。
* 🔍 **详细规范与最佳实践**，请参阅：[docs/agents/chunk_pipeline.md](docs/agents/chunk_pipeline.md)。

### 道具与物品系统架构
* 游戏中的物品（如食物、工具等纯道具）与方块系统完全解耦。右键点击放置方块或食用食物的交互逻辑采用基于上下文的多态分发机制驱动。同时通过属性解析器注入模式彻底消除了模块循环依赖。
* **重要开发契约**：如果需要新增/修改物品（如添加食物、武器或工具）、方块配置或者扩展右键交互动作（放置方块、使用道具等相关功能），**必须**先阅读此文档以确保符合多态交互与依赖注入的开发规范。
* 🔍 **详细规范与最佳实践**，请参阅：[docs/agents/item_system.md](docs/agents/item_system.md)。

## 5. 开发与编码规范 (全员必读)

在对本项目进行修改或增加功能时，必须严格遵守以下开发与编码规范：

**目录组织与架构**：新增或调整代码时，**必须将当前项目视为大型工程来进行组织**，编写高内聚、低耦合的代码，明确划分模块职责，以方便后期的高效拓展与长期维护。当代码行数接近或超过 600 行时，必须进行重构与拆分。应当将逻辑和视图拆分为更小粒度的子组件或独立的函数/工具模块，并组织在清晰、合理的目录结构中。

**状态管理**：全局状态统一通过 `src/store/useGameStore.ts` 的 slices 访问，禁止在组件层直接维护跨组件共享状态。

**样式编写规范**：本项目使用原生的 Sass 模块化开发。所有新添加的 React 组件都应当在同目录下创建对应的 `*.module.scss`，并使用 CSS 变量来继承全局色彩主题。

**TypeScript 强类型**：
* 所有新函数、新组件属性都必须编写完备的 TypeScript 类型。
* 在与 Three.js 交互时，避免使用 `any` 类型，严格使用 Three.js 的原生数学类（Vector3、Euler）和三维对象类型。

**避免硬编码与重复配置**：
* 严禁在代码中直接写魔法数字（Magic Numbers）、硬编码的颜色值、时长、物理属性等。
* 所有配置项必须存放在对应模块的域内配置文件中。
* 严禁复制粘贴相同的生成逻辑或概率计算，必须提取统一的通用算法函数与对应的配置结构。

**Lint和类型检查**：修改代码后，**必须**运行 `eslint`、`build` 和 `tsc` 校验并修复所有问题。严禁仅凭 IDE 提示就宣布任务完成。

**测试脚本同步更新与验证**：
* **测试识别**：在修改任何功能、核心逻辑（如 `src/game/` 内的核心计算）或状态管理（`src/store/`）时，必须主动检索并识别是否有对应的测试脚本或与源文件同级的 `.test.ts` 文件）。
* **同步更新**：如果修改的业务或核心逻辑已有对应测试，**必须同步更新、修复或重构对应的测试用例**，确保测试能真实并正确地覆盖最新的代码行为，严禁直接忽略或跳过受影响的现有测试。
* **新增测试**：若引入了新的核心算法（如新的噪波算法、数学计算工具、数据序列化逻辑等），必须在 `tests/` 或同级目录下编写并提交对应的单元测试。
* **执行验证**：在完成代码修改与测试脚本更新后，**必须**按以下规则在本地运行测试并确保全部通过，保证无回归缺陷（Regression）：
  * **重构场景**（如目录结构调整、模块拆分、接口变更等影响范围较广的改动）：**必须全量运行** `npm run test:run`，确保整个项目无回归；
  * **常规修改场景**（新增功能、局部 Bug 修复等）：运行**受影响模块的目录级测试** `npm run test:run -- path/to/module`；
  * **禁止单文件测试**：任何情况下禁止将测试范围缩小至单个文件，最小粒度为**模块目录**。

**临时文件管理**：所有生成的临时文件、调试脚本或中间产物必须存放到 `.temp/` 文件夹。
